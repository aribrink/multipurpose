Overview

We are required to create a versatile multi-purpose component in order to display various data derived from a WebApi backend service. All logic for data retrieval and parsing of that data should be handled inside this component and the user should be able to parameterize it using input parameters. The user should also be able to handle which data is visible and how the component is rendered. We would also like to have the capability to maintain and extend the component with ease if the need arises in the future.


Component Design


UI/UX

The component will consist of two main parts. The Header section and the Content section. In the header section, we will need to have a title and some action buttons that can be used to perform various tasks. For now, we need two actions: ADD and EDIT.


The content section is where the different views of the component will be rendered. We would like our component to have two distinctive views in its first iteration, adaptable enough however to be able to add more in the future.


Our first view will be the List View. Here our data will be rendered in sequence. Some parameterization might be needed for specific fields but the general idea will be to have a clean design that will be easy to adapt based on the component's user decisions. Each item in the list will render a dynamic amount of elements in a row. The selection will be handled by the user. (see hiddenFields, Root Component Inputs) The list should be scrollable of course and each item should be clickable for the component to be able to fetch its details from the backend service


Our second view will be the Details View. In this view, data will be rendered in the appearance of a form. For now, we just need to display the data in a two-column formation but that can potentially change so we should anticipate that in our implementation. What fields will be displayed will be dynamic and will be handled by the user with the use of a component parameter, like in the List view. (see hiddenFields, Root Component Inputs)

Sample Use
<app-multi 
    backendRoute = "/customer/001" 
    displayType = "details" 
    displayOption = "column-2"
    headerTitle = "Customer Details" 
    [actionAdd] = true 
    [actionEdit] = false 
    [hideDisabledActions] = true
    [hiddenFields] = "['id','first','zip']">
</app-multi>

Root Component Inputs
backendRoute :string
We use this parameter to instruct our component on how to retrieve our data.
Example:
"\products"
displayType :string
We use this parameter to instruct our component on how to render the retrieved data. For now our options will be list and details
Valid options:
"list" / "details"
displayOption :string
We use this parameter in order to anticipate future iterations with additional display options. For now, the only valid option will be the default "column-2" for the Details view and the user should be able to omit it.
Valid Option:
"column-2"
headerTitle :string
This is the parameter that defines the string used in the header of the component.
Examples:
"Products List", "Customer"
hiddenFields :string[]
The user can use this array to filter out fields that they wish not to be rendered. It should be an array of DisplayItem.id strings. See above in the Models section (model: DisplayItem) of the architecture diagram for details, or the API section.
Examples:
['id','first','street']
actionEdit :boolean
Will enable or disable the EDIT action of the component
Valid options:
true/false
actionAdd :boolean
Will enable or disable the ADD action of the component
Valid options:
true/false
hideDisabledActions :boolean
With this option, we can instruct the component to render the disabled actions inactive or hide them altogether.
Valid options:
true/false

WebApi Service

The WebApi service should have a single loadData method that receives a string parameter with the corresponding API endpoint. It should return a DataItem array. See above in the Models section (model: DataItem) of the architecture diagram for details.



API Explanation

Here is a brief explanation of the API that we will use and the schema of the data. This will help to understand how the data is structured and how we will consume them in our component.

The schema is designed so that all our views, list or details, can use a single method to retrieve them. The API responds with an array of elements. If our view is a list, the array will contain more than one element, otherwise just one. Each element encloses an object named displayedData that in return is an array with all the properties necessary to render our UI. More specifically, these properties are id, displayName, and value.

The id will be used by the user to filter out fields from the UI.

The displayName will be used to render the name of the property when necessary.

The value is the attribute that will be most commonly used when we want to render our data. This is the attribute we will use in the list items and the form elements.
